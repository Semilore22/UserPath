# Rule: Design System

## Token Files

- `variables.css` at the project root is the compiled source of truth for all design tokens. It is generated by `convert-tokens.js` from `colour-token.json` and `design-tokens.tokens.json`.
- Do not edit `variables.css` directly. Edit the JSON source files (`colour-token.json`, `design-tokens.tokens.json`) and re-run `convert-tokens.js` to regenerate `variables.css`.
- If a colour, type, or spacing value is not available, raise it as a design decision — do not invent a value.

## Using Tokens

- Always reference design tokens via CSS custom properties. Never hardcode hex colours, font sizes, or spacing values.

```css
/* ✅ Correct */
.button {
  background-color: var(--color-primary);
  font-size: var(--font-body-medium-font-size);
  padding: 8px 16px;
}

/* ❌ Never hardcode colours or font sizes */
.button {
  background-color: #000000;
  font-size: 14px;
}
```

Note: `padding` and `margin` use static px values because `--space-*` tokens do not yet exist in the token pipeline. See the Spacing section below. Spacing is the only measurement where px fallbacks are acceptable.

## CSS Modules

- Every component that has styles must have a corresponding `.module.css` file.
- The module file must be named identically to the component: `FlowCanvas.tsx` → `FlowCanvas.module.css`.
- Do not use global CSS classes for component-level styling.
- Do not use inline `style` props for anything other than dynamic values that cannot be expressed in CSS (e.g. computed node positions in the flow canvas).

## Component Conventions

### Primitive UI Components (`components/ui/`)
- Must be fully controlled (accept value and onChange).
- Must accept a `className` prop for external override when necessary.
- Must never contain business logic or make API calls.
- Must be accessible: use semantic HTML, include `aria-*` attributes where needed, support keyboard interaction.

### Feature Components (`components/features/`)
- May contain business logic and call hooks.
- Must not import from other feature component folders — feature components are isolated.
- May import from `components/ui/`.

## Flow Canvas Specifics

- The flow canvas renders using React Flow. Do not replace or wrap React Flow's core rendering with custom SVG.
- Node shape conventions must be enforced via node type registration — not via ad hoc inline styles:

| Node Type | Shape | CSS Class |
|---|---|---|
| `process` | Rounded rectangle | `.nodeProcess` |
| `decision` | Diamond | `.nodeDecision` |
| `error` | Rectangle with accent border | `.nodeError` |
| `terminal` | Circle | `.nodeTerminal` |

- Happy path edges must use a distinct stroke style referencing a colour token (e.g. `var(--color-primary)`). Do not hardcode the happy path colour.

## Typography

- Use only type values from `variables.css`. The naming convention is `--font-<category>-<size>-<property>`.
- Body text: `var(--font-body-medium-font-size)` / `var(--font-body-medium-font-family)` / `var(--font-body-medium-line-height)`
- Labels and UI copy: `var(--font-body-small-font-size)` (or `--font-body-medium-font-size` if small is not available)
- Headings: use `--font-headline-*` or `--font-title-*` or `--font-display-*` depending on the heading level
- All values use `px` units. Do not mix `px` and `rem`.

## Spacing

- The token pipeline does not currently define `--space-*` tokens. Use static px values for padding, margin, gap, and border-radius until spacing tokens are added to `colour-token.json` / `design-tokens.tokens.json`.
- See the Component Builder skill's CSS Module example for the pragmatic approach.
- If spacing tokens are added to the pipeline in the future, update this section and migrate existing components.

## What Not To Do

- Do not install Tailwind CSS. This project uses CSS Modules and design tokens.
- Do not install a component library (MUI, Chakra, shadcn). Build from the `components/ui/` primitives.
- Do not use `styled-components` or `emotion`.
- Do not add animations or transitions that are not token-defined.
