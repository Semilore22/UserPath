# Rule: Security

## Secret Handling

- All secrets live in `.env.local`. This file is never committed to version control.
- `DEEPSEEK_API_KEY` is **server-side only**. It must never appear in:
  - Any client component
  - Any file imported by a client component
  - Any `NEXT_PUBLIC_*` variable
  - Any log output
  - Any error message returned to the client
- If you are writing code that touches `DEEPSEEK_API_KEY`, confirm you are inside a Server Component, an API route (`route.ts`), or `lib/deepseek.ts`.
- `DATABASE_URL` follows the same rule — server-side only, never logged.

## Input Validation

- Validate all user input at **both** the client layer and the API layer. Client-side validation is UX. API-side validation is security.
- The API must never trust the client. Even if the client enforces 30–500 character limits, the API route must re-validate independently.
- Validate `flowType` against the allowed enum values before writing to the database. Reject unexpected values with a 400.
- Validate `targetUsers` as an array of non-empty strings. The Prisma schema defines it as `String[]` — there is no enum for target user values at the schema level. Reject non-array, null, or empty-string entries.
- Never pass raw user input directly into a DeepSeek prompt without sanitisation. Strip HTML tags and script patterns before constructing the prompt.

## Rate Limiting

- Rate limit checks use `lib/rate-limit.ts`. Do not inline rate limit logic in API routes.
- The IP address used for rate limiting must be extracted server-side from request headers. Never trust a client-supplied IP.
- Use `x-forwarded-for` carefully on Vercel — take only the first IP in the chain.
- Rate limit state is stored in the database (`RateLimit` table). Do not use in-memory rate limiting — it does not persist across serverless function invocations.

## API Route Security

- All write operations (POST, PATCH, DELETE) must verify `sessionId` ownership before modifying any record.
- Never expose raw database error messages in API responses. Catch errors and return a generic message with an appropriate status code.
- Never return a full database record when a subset is sufficient. Shape the response explicitly.
- Always return `Content-Type: application/json` from API routes.

## DeepSeek Prompt Security

- The system prompt is constructed server-side in `lib/deepseek.ts`. It must not be modifiable by user input.
- User input is injected into the user message only — never into the system prompt string.
- Prompt injection attempts (instructions embedded in the product description) are mitigated by off-topic detection before the API call. If the input passes off-topic detection, the system prompt's constraints handle the rest.
- Log DeepSeek API errors with a unique request ID for traceability — but never log the API key or the full prompt content in production.

## Session Security

- Session IDs are cuids generated by Prisma (`@default(cuid())`). On first contact, if the `x-session-id` header is missing or no `Session` record exists with that ID, the API route creates a new `Session` record and returns the ID in the response body. The client stores this ID in `sessionStorage` and includes it in the `x-session-id` header on all subsequent requests. See the api-route-scaffolder skill for the request/response pattern.
- Never accept a client-supplied session ID as authoritative without first verifying it corresponds to an existing `Session` record in the database.
- Sessions are anonymous at MVP. Do not store PII beyond IP address (used for rate limiting).
- IP addresses stored in the `RateLimit` table must not be returned in any client-facing API response.

## What Not To Do

- Do not log secrets, API keys, or full request bodies in any environment.
- Do not use `eval()` or `Function()` constructor anywhere in the codebase.
- Do not construct SQL queries with string interpolation — always use Prisma's parameterised query methods.
- Do not add CORS headers that allow all origins (`*`) on write endpoints.
- Do not commit `.env.local`, `.env`, or any file containing real credentials.
